# ScanOps 보안 취약점 탐지 모델 — 진행 보고 (교수님 전달 / 학습용)

작성일: 2026-07-01 · 대상: scanops-model 레포 · **이전 V11 보고(2026-06-27)를 전면 교체**

> **한 줄 요약:** 자체 파인튜닝한 **7B 보안 모델 + 다언어 taint 정적분석** 하이브리드가,
> **실제 CVE 벤치마크(CVEfixes)와 독립 벤치마크(CyberNative)에서 상용 모델 Grok-3-mini를 능가**한다
> (CVEfixes F1 67.6 vs 43.7, CyberNative F1 79.1~92.0 vs 65.5). 적대적 합성 벤치(OWASP)에서는
> Grok이 더 공격적으로 탐지해 재현율은 높으나 **안전 코드의 64%를 오탐**했고, 우리 모델은
> **오탐률을 1/3 이하로 억제**했다.
>
> **⚠️ 이전 보고 정정:** 지난 V11 보고의 "OWASP F1 88.3으로 Grok 전 지표 능가"는
> **OWASP를 학습과 평가에 동시 사용한 데이터 누수** 때문이었다. 이를 제거하고 정직하게
> zero-shot으로 재평가하니 수치가 낮아졌다. 아래는 누수를 차단한 **정직한 최신 수치**다.

## 0. 이 문서를 읽는 법
파이프라인: **데이터 → 학습 → 추론 → 그래프 검증 → 벤치마크**. 각 단계에서 무엇을/어떻게/왜
했는지, 그리고 이번 라운드(V12→V14)에서 **틀린 가설과 그 교훈**까지 정직하게 적었다.

방법론 용어:
- 트랜스포머 LLM의 **지도 파인튜닝(SFT)** — 고전 ML 아님. 최적화 = AdamW + 역전파, 손실 = 토큰 교차엔트로피.
- 효율화 = **QLoRA**(4bit 양자화 + LoRA 저랭크 어댑터). 평가 = 분류 지표(Precision/Recall/F1/오탐률/정확도).
- **데이터 누수(leakage)** = 평가셋을 학습에 섞는 것. 성능이 부풀려짐 → 이번 라운드의 최우선 교정 대상.

## 1. 데이터

### 1-1. NVD CVE (RAG용) — 792 → **8,883**로 확장·정제
- 출처: NVD REST API. 2026년 공개 CVE.
- **정제(`scanops/data/build_rag_index.py`)**: 무효/반려(`REJECT`/`DISPUTED`/`RESERVED`) 26건 + 중복 설명 225건 제거 → **8,883건** 확정. (이전 파이프라인이 반려 항목을 안 거르던 버그도 영구 수정.)
- 임베딩: `BAAI/bge-small-en-v1.5`(384d, L2정규화). 벡터DB: **Qdrant**(cosine). RAG 검색용.

### 1-2. 학습 데이터 — 264 → **3,483(V13) → 5,983(V14)**
V12의 최대 문제는 학습셋이 264개로 너무 작았던 것. 실제 코드로 대폭 확장했다.
- **V13(3,483)**: HuggingFace `cvefixes`(실제 CVE 패치 커밋의 취약/수정 코드쌍) 10개 언어 균형 추출.
- **V14(5,983)**: V13 + `CyberNative DPO`(합성 secure/insecure 쌍) 2,500 추가.
- ★**누수 차단(핵심)**: 평가셋 3종(OWASP 110 · CVEfixes held-out 141 CVE · CyberNative 154)을
  **CVE-id + 코드해시로 학습에서 완전 분리**. train/val도 CVE 단위 disjoint. 자가검증 통과(교집합 0).

## 2. QLoRA 파인튜닝 파라미터

| 항목 | V13 | V14 | 비고 |
|---|---|---|---|
| 베이스 | Qwen2.5-Coder-**7B**-Instruct | 동일 | V12에서 3B→7B 확정 |
| 양자화 | 4bit nf4 + double-quant | 동일 | 클라우드 GPU(A100/L4) |
| LoRA r / alpha | 32 / 64 | **64 / 128** | V14 용량 확대 |
| target modules | q,k,v,o_proj | + **gate,up,down_proj(MLP)** | V14 표현력↑ |

- 서빙: **Ollama ADAPTER 방식**(베이스 + 어댑터 GGUF, 병합 없음). temp=0, repeat_penalty=1.3.
- 학습곡선: V13 eval_loss 0.49 / V14 0.30(best=epoch2, epoch3은 과적합 → best 체크포인트 자동선택).

## 3. 버전 여정 (V11 → V14) — 가장 중요한 학습 포인트

| 버전 | 핵심 변경 | 결과 | 교훈 |
|---|---|---|---|
| V11 | OWASP 학습+평가 동시 | OWASP F1 88.3(착시) | **데이터 누수** — 정직한 값 아님 |
| V12 | OWASP 학습 제외(zero-shot) | 실제 CVE 재현율 1.2% | 누수 제거하니 **학습 부족(264개) 한계 노출** |
| **V13** | CVEfixes 3,483(13배) | **CVEfixes F1 67.6, Grok 능가** | **데이터 확장이 판별력을 살림** |
| V14 | +합성주입 2,500(r64+MLP) | 초정밀·저재현율(가설 빗나감) | **discrimination-pair 데이터는 재현율 역효과** |

**V13의 성과:** 실제 CVE 재현율이 V12의 1.2%에서 **62.5%로 52배** 올랐고, Grok을 능가했다.
독립 벤치(CyberNative)에서도 이겨 "특정 데이터를 외운 게 아닌 일반화"임을 확인했다.

**V14의 정직한 결과(negative result):** "합성 주입 패턴을 더 학습시키면 OWASP 재현율이 오를 것"
이라는 가설로 CyberNative 합성쌍을 추가했으나, **오히려 재현율이 떨어졌다**(OWASP 47→36%,
CVEfixes 62→54%). "같은 코드의 안전/취약 쌍" 학습은 모델을 **더 보수적(초정밀)**으로 만들어
거짓 알람은 급감(오탐률 최저 5.2%)했지만 놓치는 취약점이 늘었다. → **재현율을 올리려면
다른 접근(threshold 캘리브레이션·2단계 검증·recall 지향 손실)이 필요**하다는 것을 실험으로 확인.

## 4. 코드 그래프 (다언어 정적 taint 분석)
- 커버: Java 전용 + JS/TS 멀티파일(Neo4j) + generic 7언어(Python/PHP/Go/C#/Ruby/Node/TS).
- 판정: `analyze(code, lang) → {verdict: vuln|safe|unknown}`. **확신할 때만 vuln/safe, 애매하면 unknown→LLM 위임.**
- **결합규칙 R2(이번 개선):** 그래프는 **LLM이 놓친 취약만 보강**(재현율↑). LLM의 취약 판정은 절대 덮지 않음
  → 그래프가 틀려도 정탐 제거 0. 3벤치 튜닝(`scripts/tune_hybrid_rule.py`)에서 이전 규칙을 모두 ≥.
- **그래프는 상황의존:** LLM이 약한 OWASP엔 도움(재현율↑), LLM이 강한 CyberNative엔 오히려 노이즈.
  → 만능이 아니라 "LLM 약점 보완재". C/C++는 아직 미커버(다음 과제).

## 5. 추론 파이프라인
```
코드 입력
  → ① 파인튜닝 7B 모델이 1차 판정 (취약/안전 + CWE + CVSS)
  → ② 코드그래프 taint 검증(R2 결합) → LLM 놓친 취약 보강
  → ③ (신규 CVE 의심 시) RAG로 Qdrant 8,883 CVE 검색 — 최신성 보강 모듈
  → 최종: 취약점 + CWE + CVSS + 근거 + 수정 가이드
```
- 서빙: Ollama, GGUF(어댑터), CPU 서빙 가능. temp=0(재현가능), num_predict=200(3줄 구조화 출력).
- ※ RAG는 판별 정밀도를 낮춰(과탐 강화) **메인 탐지 경로에서 제외**, "신규 CVE 재현" 전용으로 분리 운용.

## 6. 벤치마크 결과 (정직·재현가능, temp=0, 동일 케이스 1:1)

**ScanOps = 7B 파인튜닝 + 코드그래프(R2) 하이브리드.**

| 벤치 | 시스템 | F1 | 재현율 | 오탐률 | 정확도 |
|---|---|---|---|---|---|
| **CVEfixes**(실제 CVE) | **ScanOps V13** | **67.6** | 62.5% | 23.4% | **69.4%** |
| | ScanOps V14 | 63.7 | 53.8% | **15.6%** | 68.8% |
| | Grok-3-mini | 43.7 | 32.5% | 16.9% | 57.3% |
| **CyberNative**(독립) | ScanOps V13 | 79.1 | 93.5% | 42.9% | 75.3% |
| | **ScanOps V14** | **92.0**† | 89.6% | **5.2%** | **92.2%** |
| | Grok-3-mini | 65.5 | 71.4% | 46.8% | 62.3% |
| **OWASP**(합성) | ScanOps V13 | 57.1 | 47.3% | 18.2% | 64.5% |
| | ScanOps V14 | 48.8 | 36.4% | **12.7%** | 61.8% |
| | Grok-3-mini | **70.5** | 89.1% | 63.6% | 62.7% |

†CyberNative V14는 LLM 단독 값(그래프 결합 시 86.4). 일부 in-distribution 효과 포함.

**해석:**
- **실제 CVE(CVEfixes)·독립(CyberNative)에서 ScanOps가 Grok 능가** — 가장 현실적인 두 벤치에서 우위.
- **OWASP는 Grok이 재현율 우위**지만 **오탐률 63.6%(안전 코드 절반 이상 오탐)**. ScanOps는 오탐률 13~18%로 **훨씬 정밀**.
- **V13 = 균형(재현율·일반화 우위), V14 = 초정밀(거짓 알람 최소, 재현율 낮음).** 스캐너 용도(놓치면 안 됨)엔 **V13 권장**.

## 7. 종합 결론 — 3-레이어 아키텍처
| 레이어 | 역할 | 입증 |
|---|---|---|
| **QLoRA LLM(7B)** | 취약점 탐지 + CWE | 실제 CVE·독립 벤치서 Grok 능가 |
| **코드 그래프(taint)** | LLM 놓친 취약 보강 | OWASP 재현율 +24%p(LLM 약한 구간) |
| **RAG(NVD 8,883)** | 최신 CVE 커버 | 학습 컷오프 이후 신규 취약점(정량 검증 예정) |
| **자체 호스팅** | 코드 비전송·무과금 | Ollama CPU 서빙, 외부 API 불필요 |

핵심 차별점: **코드를 외부로 보내지 않고, 작은 자체 7B 모델로 상용 모델급/이상 정확도**를 실제 CVE에서 달성.

## 8. 현재 한계 & 다음 단계 (V15)
